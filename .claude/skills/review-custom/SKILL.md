# コードレビュー

このプロジェクトのコードをレビューする。

- `$ARGUMENTS` が指定されている場合: そのファイル/ディレクトリを対象にレビューする
- `$ARGUMENTS` が空の場合: 現在のブランチの未コミット変更を対象にする

## レビュー手順

1. **変更の確認**
   - `git status` で変更されたファイルを確認
   - `git diff` でステージされた・されていない変更を確認
   - `git diff HEAD` で全変更を確認
   - 引数指定がある場合は対象ファイル/ディレクトリを直接 Read/Grep する

2. **コード品質のチェック**
   - 可読性（変数名、関数名、コメント）
   - 構造（関数の長さ、複雑度、責務の分離）
   - 重複コードの有無
   - エラーハンドリングの適切性

3. **ベストプラクティスの確認**
   - 言語・フレームワーク固有のベストプラクティス
   - デザインパターンの適切な使用
   - パフォーマンス（不要な計算、メモリ使用）
   - アクセシビリティ（該当する場合）

4. **セキュリティチェック**
   - 機密情報のハードコード（APIキー、パスワード等）
   - 入力のバリデーション
   - SQLインジェクション、XSS等の脆弱性
   - 適切な認証・認可

5. **TypeScript/型の確認**（該当する場合）
   - 適切な型定義
   - `any` の不要な使用を避けているか
   - 型安全性の確保

6. **テスト**
   - 既存テストが通るか
   - 新機能に対するテストの必要性
   - テストカバレッジ

7. **ドキュメント**
   - コードコメントの適切性
   - README等のドキュメント更新の必要性
   - 複雑なロジックの説明

## レビュー結果の報告フォーマット

### ✅ 良い点
- 良い実装や改善点をリストアップ

### ⚠️ 改善提案
- 改善が推奨される点を、重要度と理由付きでリストアップ

### 🚨 重大な問題
- セキュリティやバグなど、修正必須の問題をリストアップ

### 💡 オプション提案
- 必須ではないが検討に値する改善案

## 注意事項

- 建設的で具体的なフィードバックを提供する
- 問題を指摘するだけでなく、解決策も提案する
- コードスタイルよりロジック・セキュリティを優先する
- プロジェクトのコンテキスト（CLAUDE.md、技術スタック）を考慮する
